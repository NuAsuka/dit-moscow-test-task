# Ответы на вопросы

## 1. Почему JOIN, а не LEFT JOIN?

**LEFT JOIN** нужен, когда мы хотим **сохранить все строки** из левой таблицы, даже если в правой нет совпадений (тогда будут NULL).

В нашем запросе **LEFT JOIN не нужен**, потому что:

- Нам нужны **только те записи**, где есть совпадение по всем таблицам
- Если у песни нет артиста в `song_artists` — такая строка нам **не интересна**
- Если бы мы использовали LEFT JOIN, получили бы строки с `NULL` в поле `artist`, которые всё равно бесполезны

```sql
-- С LEFT JOIN (неправильно для этой задачи):
-- Если у песни нет артиста → artist = NULL, строка всё равно считается
-- Это исказит результат

-- С JOIN (правильно):
-- Берём только полные записи, где есть и песня, и артист
```

**Правило:** используй `JOIN` (INNER JOIN), когда тебе нужны только совпадающие записи. Используй `LEFT JOIN`, когда нужно сохранить все строки из основной таблицы.

---

## 2. Происходит ли декартово произведение?

**Нет, декартова произведения нет.** Декартово произведение возникает, когда таблицы соединяются **без условия** (`ON ...`).

В нашем запросе **все JOIN имеют условия**:

```
listening_logs → songs          ON s.song_id = ll.song_id        ✅
songs          → song_artists   ON sa.song_id = s.song_id        ✅
song_artists   → artists        ON a.artist_id = sa.artist_id    ✅
```

### Но есть **размножение строк** из-за связи one-to-many:

Если у песни **2 артиста** в `song_artists`, то одна запись из `listening_logs` превратится в **2 строки**:

```
Песня X (10 прослушиваний) × 2 артиста = 20 строк
```

Это **не ошибка**, а нормальное поведение. Именно поэтому используется:

```sql
COUNT(DISTINCT ll.song_id)  -- считаем уникальные песни, а не строки
```

### Пример:

| song_id | artist | listen_time |
|---------|--------|-------------|
| 1       | A      | 10:00       |
| 1       | B      | 10:00       | ← та же песня, второй артист
| 1       | A      | 11:00       |
| 1       | B      | 11:00       |

`COUNT(*)` = 4 ❌
`COUNT(DISTINCT song_id)` = 1 ✅ (песня одна, но засчитана обоим артистам)

### USER
есть ли в sqlite using для join

### ASSISTANT
Да, в SQLite **есть** поддержка `USING` для `JOIN`. 

Она используется как более короткая альтернатива `ON`, когда столбцы, по которым вы соединяете таблицы, **называются абсолютно одинаково** в обеих таблицах.

### Пример из ваших задач:

**Вариант с `ON` (стандартный):**
```sql
SELECT * 
FROM songs s
JOIN albums a ON a.album_id = s.album_id
```

**Вариант с `USING` (сокращенный):**
```sql
SELECT * 
FROM songs s
JOIN albums a USING (album_id)
```

###  Важная особенность `USING`:
Если вы используете `SELECT *`, то при `ON` в результате будет **два** столбца `album_id` (один из `songs`, другой из `albums`). 
А при `USING` SQLite автоматически объединит их, и столбец `album_id` в результате будет **только один**.

### Когда удобно использовать:
Это отлично подходит для ваших таблиц, где внешние ключи часто называются так же, как и первичные (например, `song_id`, `album_id`, `artist_id`, `genre_id`).

```sql
-- Можно соединить сразу несколько таблиц через USING:
SELECT *
FROM listening_logs ll
JOIN songs s USING (song_id)
JOIN albums a USING (album_id)
JOIN song_artists sa USING (song_id)
JOIN artists ar USING (artist_id)
```
