## Обеспечение качества данных (Data Quality)

В проекте реализован этап валидации и фильтрации данных (механизм Quarantine) перед их загрузкой в хранилище. 

### Реализованные проверки (Валидация)
*   **Удаление дубликатов:** Для всех источников применен метод `.drop_duplicates()`.
*   **Проверка ключевых полей (Not Null):** Строки с пустыми значениями в ID (например, `order_id`, `customer_id`) отфильтровываются.
*   **Валидация типов данных:** Даты приводятся к формату TIMESTAMP, числовые метрики (суммы, цены) проверяются на соответствие типам `int`/`float`.
*   **Бизнес-логика:** Цены и суммы проверяются на отсутствие отрицательных значений (`price >= 0`).

### Описание принятых решений

#### 1. customers.csv

1. Не все значения `full_name` имели корректный формат (присутствовали обращения "г-н", "товарищ" и т.п.). В качестве решения реализована функция очистки, удаляющая префиксы перед валидацией.
2. 281 запись содержит пустые значения в поле `phone`. Принято решение сохранить такие записи. Удаление привело бы к потере 40% данных.
3. 30 записей не содержат значений `email`, 20 записей — `city`, 16 записей — `created_at`. Указанные поля определены как необязательные (`nullable=True`), поэтому записи сохранены в очищенном датасете.

#### 2. events.xml

1. Обнаружена запись с невалидным значением `event_id = "BAD_ID"`. Запись отправлена в карантин.
2. 51 запись содержит некорректные или отсутствующие значения `event_timestamp` (включая значение `"broken-date"` и `NaN`). Поле определено как обязательное (`nullable=False`), записи отправлены в карантин.
3. 1 запись содержит пропуск в поле `customer_id`. Поле определено как обязательное, запись отправлена в карантин.

#### 3. orders.json

1. Обнаружены записи с некорректной датой `order_timestamp = "2025-99-99"` (несуществующий месяц). Записи отправлены в карантин.
2. 54 записи содержат пропуски в поле `customer_id`. Поле определено как обязательное . Записи отправлены в каранттин.

#### 4. payments.csv

1. Обнаружена запись с невалидным значением `amount = "error_amount"`.. Запись отправлена в карантин.
2. 268 записей содержат пропуски в поле `payment_method`. Поле определено как обязательное, так как способ оплаты критичен для финансовой отчётности. Записи отправлены в карантин.
3. Обнаружены записи с некорректной датой `payment_timestamp = "13/45/2025"` (несуществующий месяц). Записи отправлены в карантин.

#### 5. products.xlsx

1. Обнаружена запись с невалидным значением `product_name = "О"` (длина менее 2 символов). Применена проверка минимальной длины названия. Запись отправлена в карантин.
2. 19 записей содержат пропуски в поле `price`. Поле определено как обязательное (`nullable=False`), так как цена необходима для каталога товаров. Записи отправлены в карантин.


### Логирование проблемных записей
Все «грязные» строки, не прошедшие валидацию, автоматически исключаются из основного пайплайна и сохраняются в файлы формата csv в `data/quarantine`. Подробное логирование находится `data/etl.log`
